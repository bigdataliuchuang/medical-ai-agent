# 面试材料 — 医疗 Data Agent Runtime

## 项目一句话定位

参考 Hermes Agent 的 Runtime 架构思想，设计并实现面向医疗数仓场景的 Data Agent 原型系统。
不只是 Text-to-SQL Demo，而是把医疗元数据、指标口径、SQL 安全控制和评测体系全部串联的 Agent 工作流。

## 简历描述

**医疗 Data Agent Runtime**

项目描述：
参考 Hermes Agent 的 Runtime 架构思想，设计并实现面向医疗数仓场景的 Data Agent 原型系统，
支持基于自然语言的数据分析任务，通过医疗元数据、指标口径、字段字典和数据质量规则增强上下文，
结合工具调用实现 Text-to-SQL、SQL 安全校验、只读查询执行、数据质量检查和结果解释。

核心工作：
1. 设计五层 Agent Runtime 架构：LLM 推理层、工具调用层、记忆层、技能沉淀层、任务调度层。
2. 实现 ReAct Agent Loop，支持 Thought→Action→Observation 多步推理和工具调用，含自动修复重试。
3. 设计 Prompt Assembly，将医疗表结构、字段注释、指标口径、药品字典、诊断字典和 SQL 规范动态注入上下文。
4. 实现 Tool Registry，封装元数据查询、指标查询、SQL 校验、SQL 执行、DQ 检查等 5 个核心工具。
5. 构建 Skills 体系（8 个 SOP 文件），将抗肿瘤药物分析、诊断质量检查等医疗分析流程沉淀为可复用流程。
6. 设计 SQL 安全控制机制（7 条规则），支持只读 SQL、敏感字段拦截、LIMIT 限制、时间范围检查和审计日志。
7. 构建 Evaluation Harness，通过 30 道标准问题评估 SQL 正确率、执行成功率、口径一致性和安全合规性。
8. 实现 Session Storage（5 张 SQLite 表），完整记录每次 Agent 执行的工具调用链、SQL 审计和评测结果。

技术栈：Python、Claude API、GraphRAG、Milvus、Apache Doris、DuckDB、SQLite、sqlglot、FastAPI、pytest。

## 3 分钟项目介绍

"我做的这个项目叫医疗 Data Agent Runtime，灵感来自 Hermes Agent 的架构设计。

核心出发点是：Text-to-SQL 的真正难点不在 SQL 语法，而在于模型不知道医疗数仓的业务上下文——
比如患者怎么去重、药品分类怎么识别、诊断编码用哪个字段。

所以我没有做一个简单的"输入问题、输出 SQL"的 Demo，
而是设计了一个五层架构：

第一层是 LLM 推理层，用 ReAct 模式让模型能多步规划和工具调用；
第二层是工具调用层，封装了元数据搜索、SQL 生成、安全校验、执行和分析 5 个工具；
第三层是记忆层，用 SQLite 保存对话历史，支持多轮追问；
第四层是技能沉淀层，把成功的查询模式存起来，类似问题可以复用；
第五层是任务调度层，支持把复杂多步问题拆分后按依赖顺序执行。

在医疗数据安全方面，我设计了 7 条 SQL 安全规则，
包括高危关键词拦截、敏感字段检测（id_card、phone 等）、时间范围强制和完整审计日志。

最后我还做了一个评测体系，用 30 道标准医疗问题来评估 Agent 的 SQL 准确率、安全合规率和表选择准确率。

这样就把我的医疗数仓经验、数据治理经验、SQL 能力和 AI Agent 能力串起来了。"

## 10 个常见面试追问

### Q1: 为什么不直接用 LangChain？

答：LangChain 是通用框架，医疗场景有很多特殊需求：
- SQL 必须经过安全校验，不能直接执行模型生成的 SQL
- 元数据需要医疗专业的 schema 设计（表的口径、字段的敏感性）
- 技能沉淀需要结合医疗业务经验（抗肿瘤药物分析的 SOP）
直接用 LangChain 反而会引入很多无关复杂度。自己实现核心模块，便于面试讲清楚每一层的设计决策。

### Q2: ReAct 和 Chain-of-Thought 有什么区别？

答：Chain-of-Thought 是让模型在推理过程中分步思考，但还是一次性输出结果。
ReAct 在此基础上增加了"工具调用"闭环：
- Thought: 分析当前状态，决定下一步
- Action: 调用外部工具（搜索元数据、执行 SQL）
- Observation: 获取工具结果，继续推理
这样模型可以根据真实返回的数据调整方向，而不是凭空推断。

### Q3: SQL 安全控制为什么重要？医疗场景有哪些特殊考虑？

答：医疗数据涉及患者隐私，SQL 安全控制非常关键：
1. 只读控制：禁止 DELETE/UPDATE/DROP 等破坏性操作
2. 敏感字段拦截：id_card、phone、patient_name 等不能明文查询
3. 全量导出限制：必须有 LIMIT，防止大规模数据泄露
4. 审计日志：每次 SQL 都记录到 agent_sql_audit 表，满足合规要求
这些规则在 SqlGuard 中统一管理，新增规则只需修改配置，不影响其他模块。

### Q4: Skills 和 Memory 有什么区别？

答：
- Memory（记忆层）：保存对话历史，让模型知道"上一轮你说了什么"，支持多轮追问
- SkillStore（技能沉淀）：保存成功的查询模式（问题→SQL→答案），遇到类似问题直接参考历史经验
- Skills（技能文件）：由人工编写的 SOP 文档，描述特定场景的业务口径、推荐表和 SQL 模板

三者是互补的：Skills 是领域专家经验，SkillStore 是机器学到的经验，Memory 是当前会话上下文。

### Q5: 评测体系怎么用？评什么？

答：EvalRunner 用 30 道标准医疗问题评估 Agent：
1. SQL 语法通过率：用 sqlglot 解析，判断是否是合法 SQL
2. SQL 安全合规率：用 SqlGuard 校验，是否通过全部安全规则
3. 表选择准确率：解析 SQL 中的表名，和期望表计算 Jaccard 相似度
评测不需要真实数据库，可以在 CI 中自动运行。
目标是：每次模型或 Prompt 调整后，能量化地知道准确率是否退步了。

### Q6: Prompt Assembly 具体怎么做的？

答：系统提示词由几层叠加组成：
1. Agent 角色定义（医疗数据分析 Agent）
2. 工作流说明（5 个工具的使用顺序）
3. 匹配到的 Skill SOP（如 anti_tumor_drug_usage.md 的业务口径）
4. GraphRAG 检索到的相关表结构、指标口径
5. SQL 安全规则

这样模型在生成 SQL 前，已经有了足够的领域上下文，减少幻觉。

### Q7: 为什么用 GraphRAG 而不是普通向量检索？

答：医疗元数据之间有强依赖关系（血缘关系）：
比如 ads_drug_usage_trend 依赖 dws_tumor_drug_usage_1d，dws 依赖 dwd_order，dwd_order 依赖 dim_drug_dict。
普通向量检索只能找到语义相似的表，但不知道这些表如何关联。
GraphRAG 在向量检索基础上加了图遍历，能自动找出关联表和推荐的 Join 路径。

### Q8: DuckDB 和 Doris 在这个项目里怎么用的？

答：
- Doris：生产环境，实际医疗数仓的查询数据库，基于 MySQL 协议，支持大规模分析查询
- DuckDB：本地开发和测试环境，不需要启动服务，直接用文件，支持完整的 OLAP 语法
通过 Executor 接口抽象，切换数据库只需改配置，不影响其他代码。

### Q9: 这个系统有没有已知局限性？

答：诚实说有几个：
1. 大语言模型本身的幻觉问题，尤其是复杂 JOIN 的 SQL 可能不准
2. 当前 Skill 匹配是关键词匹配，更复杂的意图匹配需要嵌入向量
3. 评测体系用 Jaccard 相似度评估表选择，不能评估 SQL 逻辑是否真正正确
4. 多轮修复最多重试 2 次，对复杂错误可能不够
这些是下一步改进方向。

### Q10: 和别人做的 Text-to-SQL 项目有什么不同？

答：大多数 Text-to-SQL 项目只关注"能不能生成 SQL"，
我的项目额外关注：
1. 业务口径准确性（医疗数仓的患者去重、指标定义）
2. 数据安全控制（医疗隐私合规）
3. 可观测性（完整执行链路记录）
4. 技能沉淀（经验复用，不是每次从零推理）
5. 系统化评测（不只是手动测试几个例子）

这些是从医疗数据治理实际工作经验中总结出来的，不是照抄开源项目。
